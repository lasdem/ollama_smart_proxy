# v4.10 — Tool Calls & Metadata Display

## Summary

The proxy now extracts, stores, and displays tool calls, finish_reason, and token
usage from streaming responses. Multimodal content in request messages and tool
results are handled gracefully. Session fingerprinting is extended to include
tool_calls and tool_call_id for proper conversation chaining during tool use.

## Changes

### Extraction (stream_tap.py)
- New `StreamMetadata` class accumulates tool_calls, finish_reason, prompt_eval_count, eval_count
- Ollama `/api/chat`: extracts `message.tool_calls` (complete per-chunk)
- OpenAI `/v1/chat/completions`: accumulates `delta.tool_calls` by index (partial arguments concatenated)
- Ollama done chunks: extracts `done_reason`, `prompt_eval_count`, `eval_count`
- OpenAI: extracts `choices[0].finish_reason` and `usage` object
- `extract_parts_from_ndjson` emits `("tool_calls", json)` parts
- `tee_stream` on_done signature extended to `(rid, content, thinking, meta)`
- `tee_stream` fires `on_chunk` with `kind="tool_calls"` for live streaming

### Request Body (smart_proxy.py)
- `_extract_text_from_content()`: handles multimodal content arrays (text parts joined, images shown as `[image]`)
- `enqueue_request`: extracts `tools_available` (JSON array of tool names) from request body
- `enqueue_request`: handles `role: "tool"` messages for prompt_text extraction
- Session fingerprinting includes `tool_calls` and `tool_call_id` in hash computation
- Outgoing fingerprint uses `_extract_text_from_content` for non-string content
- `on_stream_done`: passes all metadata fields to `log_request`
- When response has tool calls but no text content, `response_text` shows `[Tool calls: name1, name2]`

### Storage (database.py, data_access.py)
- New columns on `RequestLog`: `tool_calls_json`, `finish_reason`, `prompt_eval_count`, `eval_count`, `tools_available`
- Auto-migration adds columns to existing databases
- `log_request` accepts and persists all new fields

### API (proxy_endpoints.py)
- `query_db` and `proxy_request_detail` return all new fields in response dicts

### Live Broadcast (live_broadcaster.py)
- `accumulated_tool_calls` tracked per active request
- `kind="tool_calls"` chunks broadcast with `full_tool_calls` payload
- New subscribers receive accumulated tool calls on connect

### Dashboard (app.js, app.css)
- **Conversation threads**: tool calls rendered as collapsible sections after assistant body (function name badge + formatted JSON arguments)
- **Tool results**: detected from `request_body` messages with `role: "tool"`, shown as collapsible blocks
- **Finish reason badges**: color-coded (green=stop, orange=tool_calls, red=length)
- **Token counts**: displayed as compact metadata on assistant messages
- **Detail modal**: shows finish_reason, token counts, tools_available in metadata table; tool calls in text view
- **History table**: tool call indicator icon and finish_reason badge in status column
- **Inline metadata**: includes finish_reason, token counts, tools_available
- **Live streaming**: `kind="tool_calls"` chunks rendered progressively

### Tests (test_stream_tap.py)
- 16 new tests covering:
  - Ollama tool_calls extraction from `/api/chat`
  - OpenAI tool_calls delta extraction and accumulation
  - Non-streaming OpenAI tool_calls
  - Metadata extraction (finish_reason, token usage) for Ollama and OpenAI
  - `StreamMetadata` accumulation and JSON serialization
  - `tee_stream` integration with metadata and tool_calls
  - OpenAI partial argument accumulation across multiple chunks

### Bug Fixes

#### Tool result display (app.js)
- Messages with `role: "tool"` were incorrectly shown as "User" messages
- Added `lastMessageIsToolResult()` detection; now renders with "Tool Result" label and dedicated `.tool` CSS class

#### Session fingerprint normalization for tool calls (smart_proxy.py)
- Conversations using tool calls were split into separate sessions in the dashboard
- Root cause: Ollama stores `tool_calls.function.arguments` as a Python dict, while OpenAI-compatible clients echo it back as a JSON string — causing different `json.dumps` output and different fingerprint hashes
- Added `_normalize_tool_calls_for_fingerprint()` which strips format-specific fields (`id`, `type`) and parses string arguments to dict
- Applied in both incoming and outgoing fingerprint computation paths
- 11 new tests in `test_session_fingerprint.py` covering normalization, format equivalence, and chaining

#### Tool-use conversation display (app.js, app.css)
- Intermediate tool-use turns (assistant with `finish_reason: tool_calls` and tool-result requests) were rendered as full-size user/assistant pairs, cluttering the conversation thread
- Reworked collapse logic to distinguish "last real user question" from "last final answer" — intermediate tool turns always default to collapsed
- Added `thread-msg-tool-turn` CSS class for compact, subdued styling (reduced padding, smaller font, lowered opacity)
- Tool-calls/tool-results `<details>` now properly hidden when their parent message is collapsed
