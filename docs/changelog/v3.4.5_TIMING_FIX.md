# v3.4.5 - Timing Calculation Fix

**Date**: 2026-02-02  
**Type**: Bug Fix

## Problem

Processing time (`processing_time_seconds`) was being calculated incorrectly, resulting in negative values in database logs. This made analytics and monitoring misleading.

### Example of the Bug
```
ID     CREATED_AT          STATUS     DUR(s)   WAIT(s)  PROC(s)  SOURCE_IP
329    2026-02-02 10:09:45 completed  0.938    12.274   -11.337  10.0.0.1
330    2026-02-02 10:09:45 completed  1.617    13.834   -12.217  10.0.0.1
```

### Root Cause

The timing calculation logic in `process_request()` was flawed:

```python
# OLD CODE (INCORRECT)
duration = time.time() - start_time           # Time spent processing
processing_time = duration - queue_wait       # WRONG: subtracting wait from process time
```

The issue was that:
- `duration` already represented the actual processing time (time from `start_time` to completion)
- `queue_wait` was the time spent waiting in queue (before `start_time`)
- Subtracting `queue_wait` from `duration` produced negative values

### Timeline Explanation

```
request.timestamp        start_time              time.time()
      |                      |                        |
      |<--- queue_wait ----->|<--- processing_time -->|
      |<-------------- total_duration ---------------->|
```

Where:
- **queue_wait** = `start_time - request.timestamp` (time waiting in queue)
- **processing_time** = `time.time() - start_time` (time actively processing)
- **total_duration** = `time.time() - request.timestamp` (total request time)
- **Relationship**: `total_duration = queue_wait + processing_time`

## Solution

Fixed the timing calculations in both success and error paths:

```python
# NEW CODE (CORRECT)
processing_time = time.time() - start_time        # Actual processing time
total_duration = time.time() - request.timestamp  # Total request time
```

### Changes Made

1. **Success path** (`smart_proxy.py:528-541`):
   - Changed `duration` to `processing_time` (directly calculated as `time.time() - start_time`)
   - Added `total_duration` calculation (from request receipt to completion)
   - Updated logging to use `total_duration` instead of `duration`

2. **Error path** (`smart_proxy.py:553-566`):
   - Applied the same fix
   - Removed unnecessary `queue_wait` recalculation
   - Simplified the timing logic

## Impact

- ✅ Processing times now show positive, accurate values
- ✅ Total duration correctly represents end-to-end request time
- ✅ Queue wait time remains accurate (calculated earlier when processing starts)
- ✅ Analytics and monitoring are now reliable

## Database Fields

The `request_logs` table tracks three timing values:

| Field                     | Description                                    | Calculation                            |
|---------------------------|------------------------------------------------|----------------------------------------|
| `duration_seconds`        | Total request time (receipt to completion)     | `time.time() - request.timestamp`      |
| `queue_wait_seconds`      | Time waiting in queue (receipt to processing)  | `start_time - request.timestamp`       |
| `processing_time_seconds` | Time actively processing (processing to done)  | `time.time() - start_time`             |

**Invariant**: `duration_seconds = queue_wait_seconds + processing_time_seconds`
