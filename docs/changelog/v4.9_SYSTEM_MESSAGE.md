# v4.9 — System Message Display

**Date:** 2026-04-09

## Problem

The proxy dashboard did not show the system message (system prompt) for chat requests. The `prompt_text` field only stored the last user message, and the system message — despite being present in the raw `request_body` — was never extracted, stored as a dedicated field, or displayed in either the Conversations or Requests tabs.

## Changes

### Backend

- **`smart_proxy.py`** — During enqueue, extract the system message from the first `messages[]` entry with `role: "system"`. The deprecated top-level `"system"` field is intentionally not extracted (Ollama ignores it in `/api/chat`). Pass `system_message` to `log_request()`.

- **`database.py`** — Added `system_message` column (`Text, nullable`) to `RequestLog` model. Auto-migration in `_add_missing_columns()` adds the column to existing databases on startup.

- **`data_access.py`** — `log_request()` accepts and persists `system_message`. Stored on create, preserved across status updates (queued → processing → completed).

- **`proxy_endpoints.py`** — Both `GET /proxy/query_db` and `GET /proxy/requests/{id}` now include `system_message` in their response. Also added `request_body` to `/proxy/requests/{id}` (previously missing).

### Dashboard

- **`app.js`** — `FIELDS_SESSION_LIST` includes `system_message` so conversations fetch it. Conversation thread renders a styled "System" block at the top when present. Request detail modal shows system message in the text view.

- **`app.css`** — Added `.thread-msg.system` styling (blue-purple left border, italic text, distinct background).

### Tests

- **`test_database.py`** — Added `TestSystemMessageStorage` class with 4 tests:
  - `test_log_request_stores_system_message`
  - `test_log_request_without_system_message`
  - `test_system_message_persists_on_update`
  - `test_system_message_column_exists`
