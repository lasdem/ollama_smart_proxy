# v3.7 - Tool Calling Support Fix

**Date:** 2026-02-06
**Status:** ✅ Complete (Updated to v3.7.1)

## Overview

Fixed "MODEL_NO_TOOLS_USED" and "Unexpected API Response" errors when using the proxy with AI coding assistants (like Kilo Code VSCode addon) that require tool/function calling capabilities. The proxy was previously dropping tool-related parameters and incorrectly formatting tool call responses.

## Problems Identified

### v3.7.0 - Initial Parameter Issue
When connecting through the proxy, clients received "MODEL_NO_TOOLS_USED" errors despite the model supporting tool calling. Direct connections to Ollama worked correctly.

**Root Cause:** The `process_request()` function only forwarded a whitelist of parameters (`model`, `stream`, `messages`, `api_base`, `timeout`) to LiteLLM, dropping all others including:
- `tools` - Tool/function definitions
- `tool_choice` - Tool selection strategy  
- `temperature`, `top_p`, `max_tokens` - Generation parameters
- Other OpenAI/Ollama parameters

### v3.7.1 - Response Format Issues
After fixing parameter passthrough, users reported "The language model did not provide any assistant messages" error.

**Root Causes:**
1. Response formatting didn't preserve `tool_calls` in the message structure
2. Content was accessed directly (crashes when `None` during tool calls)
3. Streaming default was incorrect for OpenAI endpoints (should be `False`, was `True`)
4. Logging failed when responses contained tool calls instead of text

## Solutions Implemented

### v3.7.0 - Parameter Passthrough

**Location:** `src/smart_proxy.py` - `process_request()` function

```python
# 2b. Passthrough all other parameters (tools, temperature, etc.)
# Exclude parameters we've already handled explicitly
handled_params = {'model', 'stream', 'messages', 'prompt'}
for key, value in request.body.items():
    if key not in handled_params:
        req_kwargs[key] = value
```

### v3.7.1 - Response Format & Streaming Fixes

**1. Preserve tool_calls in responses** (`format_output()` function)

**Non-streaming responses:**
```python
# Extract full message (including tool_calls if present)
message = response.choices[0].message
full_content = message.content or ""  # Content can be None when tool_calls are used

# For Ollama Chat format
message_dict = {"role": "assistant", "content": full_content}
if hasattr(message, 'tool_calls') and message.tool_calls:
    message_dict["tool_calls"] = [tc.model_dump() for tc in message.tool_calls]
```

**Streaming responses:**
```python
# Extract delta information (including tool_calls chunks)
delta = chunk.choices[0].delta
content = delta.content or ""

# For Ollama Chat format
message_dict = {"role": "assistant", "content": content}
if hasattr(delta, 'tool_calls') and delta.tool_calls:
    message_dict["tool_calls"] = [tc.model_dump() for tc in delta.tool_calls]
```

**2. Fix streaming defaults** (`enqueue_request()` function)
```python
# Determine stream default based on endpoint type
# OpenAI endpoints default to False, Ollama endpoints default to True
if endpoint_type in [EndpointType.OPENAI_CHAT, EndpointType.OPENAI_LEGACY]:
    should_stream = body.get('stream', False)
else:
    should_stream = body.get('stream', True)
```

**3. Improve logging for tool calls** (`process_request()` function)
```python
if hasattr(message, 'tool_calls') and message.tool_calls:
    # Log tool calls instead of content
    tool_names = [tc.function.name for tc in message.tool_calls]
    response_text = f"[TOOL_CALLS: {', '.join(tool_names)}]"
else:
    response_text = message.content
```

## Impact

**Now Fully Supported:**
- ✅ Tool/function calling parameters (`tools`, `tool_choice`)
- ✅ Tool call responses (both streaming and non-streaming)
- ✅ Multiple simultaneous tool calls
- ✅ Empty/None content when tools are used
- ✅ Correct streaming defaults per endpoint type
- ✅ Tool call logging for analytics
- ✅ Generation parameters (`temperature`, `top_p`, `top_k`, `max_tokens`)
- ✅ Control parameters (`stop`, `seed`, `frequency_penalty`, `presence_penalty`)
- ✅ Response formatting (`response_format`)
- ✅ Any future OpenAI/Ollama parameters

**Safety Maintained:**
- `litellm.drop_params = True` remains enabled
- LiteLLM will safely drop any parameters unsupported by the backend
- No breaking changes to existing functionality

## Additional Considerations Addressed

The v3.7.1 update also addresses several edge cases that could affect AI coding assistants:

1. **Endpoint-specific defaults** - OpenAI clients expect non-streaming by default
2. **Null content handling** - Tool responses often have `null` content
3. **Streaming tool calls** - Tool calls can arrive in delta chunks
4. **Analytics compatibility** - Database logging handles tool calls gracefully
5. **Multi-tool responses** - Models can call multiple functions simultaneously
