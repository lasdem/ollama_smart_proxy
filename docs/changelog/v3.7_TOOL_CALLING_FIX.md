# v3.7 - Tool Calling Support Fix

**Date:** 2026-02-06
**Status:** ✅ Complete

## Overview

Fixed "MODEL_NO_TOOLS_USED" error when using the proxy with AI coding assistants (like Kilo Code VSCode addon) that require tool/function calling capabilities. The proxy was previously dropping tool-related parameters from requests.

## Problem

When connecting through the proxy, clients received "MODEL_NO_TOOLS_USED" errors despite the model supporting tool calling. Direct connections to Ollama worked correctly.

**Root Cause:** The `process_request()` function only forwarded a whitelist of parameters (`model`, `stream`, `messages`, `api_base`, `timeout`) to LiteLLM, dropping all others including:
- `tools` - Tool/function definitions
- `tool_choice` - Tool selection strategy  
- `temperature`, `top_p`, `max_tokens` - Generation parameters
- Other OpenAI/Ollama parameters

## Solution

Implemented parameter passthrough that forwards **all** request body parameters to LiteLLM except those explicitly handled by the proxy.

**Location:** `src/smart_proxy.py` - `process_request()` function

**Changes:**
```python
# 2b. Passthrough all other parameters (tools, temperature, etc.)
# Exclude parameters we've already handled explicitly
handled_params = {'model', 'stream', 'messages', 'prompt'}
for key, value in request.body.items():
    if key not in handled_params:
        req_kwargs[key] = value
```

**Excluded from passthrough:**
- `model` - Normalized and prefixed with "ollama/"
- `stream` - Explicitly handled
- `messages` - Converted from prompt or passed through for chat endpoints
- `prompt` - Converted to messages format for LiteLLM

## Impact

**Now Supported:**
- ✅ Tool/function calling (`tools`, `tool_choice`)
- ✅ Generation parameters (`temperature`, `top_p`, `top_k`, `max_tokens`)
- ✅ Control parameters (`stop`, `seed`, `frequency_penalty`, `presence_penalty`)
- ✅ Response formatting (`response_format`)
- ✅ Any future OpenAI/Ollama parameters

**Safety:**
- `litellm.drop_params = True` remains enabled
- LiteLLM will safely drop any parameters unsupported by the backend
- No breaking changes to existing functionality
