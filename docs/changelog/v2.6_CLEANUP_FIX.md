# Version 2.6 - Fixed recently_started_models Cleanup
**Date**: 2025-12-19 13:40

## 🐛 Bug Fixed

### Issue: Stale "loaded=TRUE" Status

**Symptoms**:
- Models showing `loaded=TRUE` when they were actually unloaded from VRAM
- `recently_started_models` set never cleaned up between test runs
- Priority calculation using stale data

**Example**:
```
# All models were UNLOADED by Ollama
📤 Processing: gemma3 (loaded=TRUE ❌)  # Should be FALSE!
📤 Processing: llama3.3 (loaded=TRUE ❌)  # Should be FALSE!
```

---

## 🔧 Fix Applied

### Change: Always Clean Up After Request Completes

**Before (v2.5)**:
```python
def remove_request(self, ip: str, model_name: str):
    # ... decrement counters ...
    
    # Keep model in recently_started if still in currently_loaded
    normalized = self._normalize_model_name(model_name)
    if normalized not in self.vram_monitor.currently_loaded:
        self.recently_started_models.discard(normalized)
```

**After (v2.6)**:
```python
def remove_request(self, ip: str, model_name: str):
    # ... decrement counters ...
    
    # Remove from recently_started after request completes
    # It will be re-added if another request for same model starts
    normalized = self._normalize_model_name(model_name)
    self.recently_started_models.discard(normalized)  # ← Always remove!
```

**Reason**: 
- Model will be re-added when next request starts
- Prevents stale data from previous runs
- Cleaner state management

## 📊 Impact

### Before v2.6:
```
# Test 1: gemma3 request completes → stays in recently_started
# 5 minutes later...

```

### After v2.6:
```

# Then: Added back to recently_started when processing starts
```
