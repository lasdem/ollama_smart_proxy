# v4.5 — Streaming lifecycle, DB indexes, dashboard load

**Date:** 2026-04-03

## Summary

- **Streaming:** `active_requests`, `RequestTracker.active_request_count`, and live broadcaster cleanup now run when the upstream **body stream finishes** (in `tee_stream` `on_done`), not when `StreamingResponse` is returned. Fixes parallel-limit semantics for long streams, restores correct live chunk broadcasting, and removes the spurious `request_completed(..., "error")` on success paths.
- **`stream_tap`:** Async `on_done` callbacks are **awaited** so DB log / broadcaster / slot release complete before the response wrapper closes the HTTP client.
- **httpx:** `AsyncClient` and response stream are closed in a **`streaming_body()`** wrapper after iteration completes.
- **Database:** New composite indexes `ix_tsrecv_tcomp` (timestamp_received, timestamp_completed) and `ix_status_tsrecv` (status, timestamp_received) for typical `query_db` patterns; migration via `_add_missing_columns`.
- **Dashboard:** WebSocket-driven home refresh throttled (~2.5s min interval, Home tab only); `loadSessions` on `request_started` only when Conversations tab is active or a pending auto-open; O(1) assistant row map for live chunks; conversations query limit capped at 500.
- **Config:** `.env.example` documents `LOG_RETENTION_DAYS` and `ANALYTICS_CACHE_TTL`.
