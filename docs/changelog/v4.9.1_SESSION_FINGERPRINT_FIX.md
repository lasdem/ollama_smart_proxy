# v4.9.1 — Session Fingerprint Normalization

**Date:** 2026-04-09

## Problem

Multi-turn conversations sent as separate HTTP requests with full message history (standard OpenAI-style, as used by the Feedzai Assistant and similar integrations) were not being grouped into a single session in the Conversations tab. Each request appeared as a separate conversation.

The content-based session chaining computes a SHA-256 fingerprint of all messages (including the assistant response) when a request completes (`outgoing_fp`), then matches it against the prefix of the next request's messages (`incoming_fp`). Any byte-level difference between the stream-accumulated assistant content and the content the client echoes back in the next request's history breaks the chain.

Common causes of mismatch:
- Trailing newline in stream-accumulated content that the client strips
- Extra whitespace or different newline handling
- Whitespace normalization by the client before echoing back

## Changes

### Backend

- **`smart_proxy.py`** — Added `_normalize_for_fingerprint()` helper that strips leading/trailing whitespace and collapses internal whitespace runs to single spaces. Applied to both fingerprint computation sites:
  - **Incoming fingerprint** (in `enqueue_request`): message content is normalized before hashing the prefix
  - **Outgoing fingerprint** (in `on_stream_done`): both the original request messages and the stream-accumulated assistant response are normalized before hashing

### Tests

- **`test_session_fingerprint.py`** — New test file with 14 tests:
  - `TestNormalizeForFingerprint` (7 tests): unit tests for the normalization helper covering edge cases (empty, whitespace-only, newlines/tabs, already normalized, content preservation)
  - `TestFingerprintChaining` (7 tests): end-to-end fingerprint matching simulations covering exact match, trailing newline, extra internal whitespace, system message whitespace, multiple system messages (Feedzai-style), negative test (different content must not chain), and three-turn chain verification

## Risk Assessment

The normalization is applied only to the fingerprint computation, not to stored or forwarded content. The risk of false-positive session matches is minimal because fingerprints include the full message history (all roles and all content) — two conversations would need identical system prompts AND identical user/assistant exchanges to collide.
