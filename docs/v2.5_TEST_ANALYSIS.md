# v2.5 Test Analysis - recently_started_models Bug
**Date**: 2025-12-19 13:40
**Issue**: Models showing loaded=TRUE when they're actually unloaded

## 🐛 Bug Found in Test

### Test Setup:
- All models UNLOADED (Ollama had unloaded them)
- VRAM history populated (models used before)
- Sent 3 concurrent requests: gemma3, llama3.3, mistral

### Expected Behavior:
```
📤 Processing: gemma3 (loaded=FALSE) - Not in VRAM yet
📤 Processing: llama3.3 (loaded=FALSE) - Not in VRAM yet  
📤 Processing: mistral (loaded=FALSE) - Not in VRAM yet
```

### Actual Behavior:
```
📤 Processing: gemma3 (priority=-100, VRAM: 6.3GB, loaded=TRUE ❌)
📤 Processing: llama3.3 (priority=-90, VRAM: 70.7GB, loaded=TRUE ❌)
📤 Processing: mistral (priority=70, VRAM: 22.2GB, loaded=FALSE ✅)
```

**gemma3 and llama3.3 showed loaded=TRUE when they were NOT in VRAM!**

---

## 🔍 Root Cause Analysis

### Why did this happen?

1. **gemma3 request arrives** → `tracker.add_request()` called → added to `recently_started_models`
2. **llama3.3 request arrives** → added to `recently_started_models`
3. **mistral request arrives** → added to `recently_started_models`

4. **Priority calculation** (while requests are queuing):
   - gemma3: `is_model_loaded()` checks `recently_started_models` → **TRUE** ✅
   - llama3.3: `is_model_loaded()` checks `recently_started_models` → **TRUE** ✅
   - mistral: `is_model_loaded()` checks `recently_started_models` → **TRUE** ✅

But wait... mistral showed loaded=FALSE. Why?

**Answer**: The code calls `tracker.add_request()` INSIDE the lock AFTER popping from queue:
```python
async with queue_lock:
    # ... select request ...
    request_queue.pop(idx)  # Remove from queue
    # ... logging ...
    tracker.add_request()   # Add to recently_started AFTER selection
```

So:
1. gemma3 selected → added to recently_started
2. llama3.3 selected → sees gemma3 in recently_started → loaded=TRUE
3. mistral selected → sees gemma3 + llama3.3 in recently_started

**But**: gemma3 and llama3.3 were NOT in `currently_loaded` (Ollama had unloaded them).

The bug: `recently_started_models` persisted from PREVIOUS test runs and was never cleaned up!

---

## 🔧 Fix Applied in v2.6

### Change 1: Clean up immediately after request completes
```python
def remove_request(self, ip: str, model_name: str):
    # OLD: Only remove if not in currently_loaded
    if normalized not in self.vram_monitor.currently_loaded:
        self.recently_started_models.discard(normalized)
    
    # NEW: Always remove after request completes
    self.recently_started_models.discard(normalized)
```

**Reason**: The model will be re-added when next request starts. This prevents stale data.

### Change 2: Added cleanup method
```python
def cleanup_stale_models(self):
 """Remove models from recently_started that are not actually loaded"""
    currently_loaded_set = set(self.vram_monitor.currently_loaded.keys())
    self.recently_started_models = self.recently_started_models & currently_loaded_set
```

**Usage**: Can be called periodically to sync state.

## 📊 Expected Behavior After v2.6

**Priority calculation**:
- gemma3: +100 (swap) + 0 (ip) + 0 (wait) + 0 (rate, first) = **100**
- llama3.3: **+300 (large swap!)** + 10 (ip) + 0 (wait) + 0 = **310** 
- mistral: +100 (swap) + 20 (ip) + 0 (wait) + 0 = **120**

**Processing order**: gemma3 (100) → mistral (120) → llama3.3 (310) ✅
